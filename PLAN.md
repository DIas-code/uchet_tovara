# План разработки приложения «Учёт товара»

> Мультитенантное веб-приложение учёта одежды для нескольких бутиков.
> Всё на русском, с заделом под другие языки. Бесплатный хостинг. Изоляция данных между бутиками.

---

## 1. Выбор стека (одно решение)

| Слой | Технология | Почему именно это |
|------|-----------|-------------------|
| **Фреймворк** | **Next.js 15 (App Router) + TypeScript** | Один проект = и фронт, и бэкенд (API routes / server actions). Не нужно держать два сервера. Бесплатно деплоится на Vercel. |
| **База данных + Auth** | **Supabase** (PostgreSQL + Auth + Row Level Security) | Ключевое решение. RLS даёт изоляцию бутиков **на уровне БД**, а не только в коде — один бутик физически не сможет прочитать данные другого. Логин/пароль встроены. Бесплатный тариф. |
| **UI** | **Tailwind CSS + shadcn/ui** | Быстро, аккуратно, адаптивно (мобильная версия из коробки). |
| **Локализация** | **next-intl** | Русский по умолчанию, добавление языка = один JSON-файл. |
| **Хостинг** | **Vercel** (фронт+бэк) + **Supabase** (БД) | Обе платформы бесплатны на старте. |
| **Бэкапы** | `pg_dump` через **GitHub Actions** по расписанию | Бесплатно, автономно, копии вне продакшена. |

**Почему НЕ Node+Express+SQLite из отчёта:**
- Vercel/бесплатные хостинги — serverless, у них нет постоянного диска → файл SQLite будет теряться.
- SQLite не даёт нормальной изоляции нескольких арендаторов и одновременного доступа.
- Пришлось бы отдельно поднимать сервер, БД, auth, бэкапы — больше работы и денег.

Supabase закрывает БД + вход + изоляцию + бэкапы одним сервисом. Это прямой ответ на твоё требование «логины связаны с бутиками, чужой не вмешается».

## 2. Архитектура мультитенантности (изоляция бутиков)

Модель: **shared database, shared schema, tenant_id + RLS**. У каждой строки есть `boutique_id`, а Postgres RLS сам не пускает пользователя к чужим строкам.

### Таблицы

```
boutiques        (id, name, created_at)                 -- бутики
profiles         (id → auth.users, boutique_id, role)   -- связь юзер↔бутик + роль
products         (id, boutique_id, code, name, category, stock, sale_price, purchase_price, ...)
movements        (id, boutique_id, product_id, type, quantity, date, created_by)
```

- `auth.users` — встроенная таблица Supabase (логин/пароль/сессии).
- `profiles` связывает пользователя с **одним** бутиком и хранит роль (`admin` / `seller`).
- Уникальность кода товара — **в рамках бутика**: `UNIQUE (boutique_id, code)`. У разных бутиков могут быть одинаковые коды.

### Как работает изоляция и автозаполнение

1. Пользователь входит → Supabase знает его `user_id`.
2. Функция `current_boutique_id()` достаёт `boutique_id` из его `profiles`.
3. **RLS-политика** на `products` и `movements`:
   ```sql
   USING (boutique_id = current_boutique_id())          -- видит только свой бутик
   WITH CHECK (boutique_id = current_boutique_id())      -- пишет только в свой бутик
   ```
4. **Автозаполнение `boutique_id`** при вводе товара — через `DEFAULT current_boutique_id()` на колонке + триггер-страховка. То есть при добавлении товара бутик подставляется сам, вручную его не вводят и подделать нельзя.

Это ровно то, что ты просил: логин привязан к бутику, `boutique_id` заполняется автоматически, чужой бутик вмешаться не может — проверка на уровне БД, не только в UI.

### Роли (из отчёта, оставляем)
- **admin** — всё внутри своего бутика (добавлять/удалять товары, менять цены, отчёты, приглашать продавцов).
- **seller** — приход/продажа/возврат, просмотр остатков; не может удалять товары и менять цены.

## 4. Этапы разработки

### Этап 0 — Каркас (0.5 дня)
- Создать Next.js проект (TypeScript, Tailwind, shadcn/ui, next-intl).
- Завести проект в Supabase, подключить ключи.
- Настроить `messages/ru.json`.

### Этап 1 — БД и мультитенантность (1–2 дня) ⭐ ядро
- Миграции: `boutiques`, `profiles`, `products`, `movements`.
- Функция `current_boutique_id()`.
- RLS-политики на все таблицы + `DEFAULT`/триггер для авто-`boutique_id`.
- Проверка: два тестовых бутика не видят данные друг друга.

### Этап 2 — Аутентификация (1 день)
- Экран входа/регистрации (Supabase Auth).
- При регистрации первого пользователя — создаётся бутик, юзер = admin.
- Приглашение продавца в существующий бутик.

### Этап 3 — Товары CRUD (2 дня) ⭐ ядро
- Список товаров (код, категория, остаток, цена) + поиск и фильтры.
- Форма добавления/редактирования (`boutique_id` подставляется сам).
- Удаление только при `stock = 0`.

### Этап 4 — Движения: приход/продажа/возврат (2 дня) ⭐ ядро
- Операции меняют `stock` в транзакции.
- Запрет продажи больше остатка (`stock` не уходит в минус).
- История операций по товару.

### Этап 5 — Отчёты (1–2 дня)
- Остатки (`stock > 0`), продажи/приход за период.
- Экспорт CSV.

### Этап 6 — Полировка и мобильная версия (1–2 дня)
- Адаптив, крупные кнопки, нижняя панель на мобильном.
- Заготовки `kk.json` / `en.json`.

**Ядро приложения = этапы 0–4.** Хостинг и бэкапы делаем ПОСЛЕ ядра (этапы ниже).

## 6. Надёжность БД и бэкапы (после ядра)

- **Автобэкапы Supabase** — есть точечное восстановление (на платном тарифе; на бесплатном — ручные/по расписанию).
- **Свой бэкап по расписанию:** GitHub Action раз в сутки делает `pg_dump` и кладёт дамп в приватный репозиторий/облако (копия вне продакшена).
- **Тест восстановления** — раз в период проверяем, что дамп разворачивается.
- **Целостность остатков:** периодическая сверка `stock` = Σ(приход+возврат) − Σ(продажи) по `movements` (как в тестах отчёта).
- Все операции с остатком — в транзакциях, чтобы не было гонок.
