# Продолжение проекта — полный контекст

Этот файл — «передача дел»: всё, что нужно, чтобы продолжить проект, даже если
переписка удалена. **Секретов и паролей здесь нет** (репозиторий публичный).

---

## 1. Что это и где живёт

- **Приложение:** мультибутик-учёт одежды. Несколько бутиков, у каждого свой
  вход, свои товары/категории, данные изолированы.
- **Прод:** https://uchet-tovara-e1u1.vercel.app
- **Код:** https://github.com/DIas-code/uchet_tovara (**публичный** — секреты в него не коммитим)
- **Хостинг:** Vercel (приложение) + Supabase (Postgres + Auth). Всё на бесплатных тарифах.
- **Рабочая папка:** `C:\Users\Dias\Desktop\Учет товара`

## 2. Запуск локально

```bash
npm install
# создать .env.local по образцу .env.example (ключи из Supabase)
npm run dev        # http://localhost:3000
npm run build      # проверка сборки/типов
```

Node 24, npm 11. Windows.

## 3. Секреты и доступы (без значений)

- `.env.local` (в git НЕ попадает) — 3 ключа Supabase:
  `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`,
  `SUPABASE_SERVICE_ROLE_KEY`. Образец — `.env.example`.
- Эти же 3 ключа заданы в настройках проекта Vercel (Environment Variables).
- **Логины/пароли бутиков** храните отдельно (у себя / в менеджере паролей).
  Логины: `Zauresh` (рабочий), `boutique1`, `boutique2` (тестовые).
- Пароль БД (для бэкапа) — в Supabase → Project Settings → Database.

## 4. Стек и важные технические особенности

- **Next.js 16** (App Router, Turbopack), React 19, TypeScript, Tailwind v4.
- **shadcn/ui на базе Base UI** (`@base-ui/react`), НЕ Radix! API отличается:
  вместо `asChild` — проп `render={<Элемент/>}` (Dialog, Button и т.д.).
- **next-intl без i18n-роутинга:** язык в cookie `locale`, дефолт `ru`,
  файлы `messages/{ru,kk,en}.json`. Локаль в `src/i18n/request.ts`.
- **Next 16:** middleware называется `src/proxy.ts` (там защита маршрутов +
  refresh сессии Supabase).
- **Фон строго белый, light-only** — тёмная тема удалена из `globals.css`
  (требование заказчика).
- **Supabase-клиенты:** `src/lib/supabase/{client,server,middleware}.ts`
  (пакет `@supabase/ssr`).

## 5. База данных

Таблицы (`supabase/migrations/`):
- `boutiques` — бутики (арендаторы).
- `profiles` — связь auth.users → бутик + роль (`admin`/`seller`).
- `products` — товары (`boutique_id`, `code`, `name`, `category_id`, `stock`,
  `sale_price`, `purchase_price`). Уникальность `(boutique_id, code)`.
- `movements` — движения (`приход`/`продажа`/`возврат`), пересчитывают `stock`.
- `categories` — категории, уникальность `(boutique_id, name)`.

Ключевое:
- Функция `current_boutique_id()` — id бутика текущего пользователя.
- **RLS** на всех таблицах: пользователь видит/меняет только строки своего
  бутика. `boutique_id` подставляется автоматически (DEFAULT).
- Триггер `apply_movement()` — пересчёт остатка в транзакции, запрет минуса.

**Применённые миграции:** 0001–0005. Новые миграции применяются вручную:
Supabase → SQL Editor → вставить SQL файла → Run. (DDL через ключи API нельзя —
только через SQL Editor или строку подключения к БД.)

## 6. Вход (по логину, не email)

Форма спрашивает **логин** и пароль. Supabase логинит по email, поэтому логин
превращается в служебный адрес `логин@uchet.local` (`src/lib/username.ts`).
Пользователь email не видит. Регистр логина не важен.

Новый бутик/логин заводится вручную — см. `docs/create-boutique.md`
(в Supabase создать пользователя с email `<логин>@uchet.local`, затем SQL —
строки в `boutiques` и `profiles`).

## 7. Что реализовано

- Вход по логину, изоляция бутиков (RLS), роли admin/seller.
- Главная со статистикой (товаров, единиц, стоимость склада, нет на остатке).
- Товары: список (таблица на ПК, **карточки на телефоне**), поиск, фильтр
  «в наличии», CRUD, удаление только при нулевом остатке.
- Категории: отдельная страница управления, выпадающий список в форме товара,
  плашки-фильтры и группировка.
- Движения: приход/продажа/возврат — кнопка «Операция» **в строке товара** и на
  карточке товара; история операций; запрет ухода в минус.
- Отчёты: сводка за период + таблица операций.
- **Экспорт CSV** (товары, операции) и **импорт CSV** (товары, админ).
- Валюта ₸, i18n (ru рабочий, kk/en заготовки), 404, скелеты загрузки.

## 8. Деплой

Ничего вручную деплоить не нужно. Env-переменные заданы в Vercel.

## 9. Надёжность и бэкапы

- `.github/workflows/keep-alive.yml` — пинг прода раз в сутки (чтобы Supabase не
  засыпал). Работает без секретов.
- `.github/workflows/backup.yml` — ежедневный дамп данных в артефакт. **Требует
  один секрет `SUPABASE_DB_URL`** в GitHub (см. `docs/backups.md`) — ещё не добавлен.

## 10. Как тестировать

- **Вход через браузер вводит пользователь вручную** (ИИ по правилам не набирает
  пароли; на localhost сессия часто слетает).
- Поэтому многое проверялось **скриптами на Node** с service_role-ключом и
  anon-ключом (вход как бутик, вставки/выборки, проверка RLS/триггеров).
  Такие скрипты создавались временно в корне (`_*.mjs`) и удалялись после.
- `npm run build` — проверка типов и сборки перед пушем.

## 11. Ограничения и идеи на будущее

- Управление пользователями/бутиками — пока вручную в Supabase (можно сделать
  страницу в приложении).
- Импорт CSV только добавляет новые товары (нет «обновить по коду»).
- Тестовые данные (`A-001`, `B-001`, `UI-100`) можно удалить.
- Допереводить `kk`/`en`.
- Бэкап включить (добавить секрет `SUPABASE_DB_URL`).
- **Репозиторий публичный** — если не нужно, сделать приватным в настройках GitHub.
